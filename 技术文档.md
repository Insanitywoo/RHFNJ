# RHFNJ 技术文档

## 1. 这份文档是做什么的

这份文档的目标，不是只解释“这个项目现在是什么”，而是带你从一个能跑的原型，逐步走到一个可以长期开发、可以部署上线、可以持续迭代的完整产品。

你目前没有系统开发经验，所以后续我们合作时会遵循一个原则：

- 先把方向想清楚，再动手做
- 先做最重要的主链路，再补优化
- 每一步尽量只引入必要的技术，不为了“高级”而复杂化
- 每做完一阶段，都保证项目仍然能运行、能测试、能继续改

这个项目本质上是一个：

- 面向论文/文档问答的 RAG 系统
- 支持 PDF 上传、解析、向量检索、引用回答、流式对话
- 有前端交互界面
- 有后端 API
- 后续可以扩展成多用户、多文档库、可部署的正式产品


## 2. 当前项目现状

你当前仓库已经有一个基础原型：

- 后端：FastAPI
- 前端：Vue 3 + Vite + Element Plus
- 检索：Chroma
- Embedding：本地 HuggingFace 模型
- 大模型调用：DeepSeek API
- 数据源：PDF 文件

当前已经具备的能力：

- 上传 PDF
- 解析 PDF
- 将 PDF 切块后写入向量库
- 前端发起问题
- 后端进行相似度检索
- 调用大模型生成带引用的流式回答
- 前端展示 PDF 和聊天结果

但是当前仍然属于“原型阶段”，还没有达到“完整产品”的标准。主要缺口有：

- 项目结构还不够稳定
- 测试不完整，且已有测试与现有代码不完全一致
- 依赖没有完全声明清楚
- 数据库层还没真正建立业务模型
- 文档管理逻辑还比较粗糙
- 缺少用户系统、任务状态、文档元数据管理
- 缺少部署规范、日志规范、配置规范
- 缺少 Docker 化与线上运行方案


## 3. 这个项目最终应该长成什么样

我建议把这个项目定位成一个：

**面向研究场景的智能文档问答平台**

未来完整产品应具备以下模块：

1. 文档管理

## 5. 前端怎么做

### 5.1 推荐技术栈

- Vue 3
- TypeScript
- Vite
- Element Plus
- Axios
- Vue Router
- Pinia

### 5.2 为什么这样选

#### Vue 3
你当前项目已经用了 Vue 3，继续沿用最省成本。Vue 对新手相对友好，组件组织清晰，适合逐步搭建后台类产品。

#### TypeScript
前期会觉得麻烦一点，但长期收益非常大：

- 更少的低级错误
- 接口字段更清晰
- 更适合前后端联调
- 项目越来越大时更稳

#### Vite
开发体验好，启动快，构建快，适合现代前端项目。

#### Element Plus
你现在已经在用，适合管理后台、工具平台、研究型应用。现成组件多，能让我们把精力放在业务而不是重复造轮子。

#### Axios
请求封装成熟，后续更方便统一处理 baseURL、token、错误拦截。

#### Vue Router
后续一旦出现多个页面，比如：

- 登录页
- 文档管理页
- 知识库页
- 聊天页
- 系统设置页

就必须有路由。

#### Pinia
用于前端全局状态管理，比如：

- 当前登录用户
- 当前知识库
- 当前选中文档
- 接口请求状态
- UI 配置


### 5.3 前端推荐模块结构

后续建议整理成：

- `src/pages/`
  - 页面级组件
- `src/components/`
  - 可复用组件
- `src/api/`
  - 所有后端接口请求
- `src/stores/`
  - Pinia 状态
- `src/types/`
  - TS 类型定义
- `src/utils/`
  - 工具函数
- `src/assets/`
  - 静态资源


### 5.4 前端后续要做的核心页面

第一阶段：

- 文档问答主页面
- 文档上传入口
- 文档列表
- PDF 预览

第二阶段：

- 登录/注册页面
- 用户个人空间
- 知识库管理页
- 系统设置页

第三阶段：

- 问答历史记录页
- 多文档对比分析页
- 摘要与综述页


## 6. 后端怎么做

### 6.1 推荐技术栈

- FastAPI
- Pydantic
- SQLAlchemy 2.x
- Alembic
- Celery 或 RQ
- Redis
- LangChain（适量使用）
- Chroma 或后续替换为更成熟向量数据库

### 6.2 为什么这样选

#### FastAPI
适合 Python API 服务开发：

- 写起来清晰
- 自动生成文档
- 对异步支持好
- 很适合 AI 应用和工具型产品

#### Pydantic
用于请求与响应的数据校验，非常适合你这种“接口很多、字段很多”的项目。

#### SQLAlchemy
负责传统业务数据库读写，处理：

- 用户
- 文档元信息
- 聊天记录
- 任务状态
- 知识库配置

#### Alembic
负责数据库迁移。以后你改了数据表结构，不能靠手工改数据库，必须靠迁移管理。

#### Redis
可以用于：

- 缓存
- 任务队列
- 会话临时状态
- 限流

#### Celery 或 RQ
PDF 解析、向量化、重建索引这种任务可能很慢，不适合一直堵塞 HTTP 请求。应该放到后台任务中异步执行。

如果后面你想尽量简单，前期可以先用 FastAPI BackgroundTasks 过渡，但正式产品更建议 Redis + Celery。


### 6.3 后端推荐分层

后端建议逐步整理成：

- `app/api/`
  - 路由层
- `app/schemas/`
  - 请求响应模型
- `app/models/`
  - 数据库 ORM 模型
- `app/services/`
  - 业务逻辑
- `app/repositories/`
  - 数据访问层
- `app/core/`
  - 配置、日志、安全、数据库初始化
- `app/tasks/`
  - 异步任务

### 6.4 为什么要分层

因为后面项目会越来越复杂。  
如果所有逻辑都堆在路由里，会很快失控。

分层的好处：

- 好维护
- 好测试
- 好复用
- 出问题容易定位


## 7. 数据库应该怎么处理

这个项目其实有两种“数据库”。

### 7.1 关系型数据库

推荐：

- 开发阶段：SQLite 或 PostgreSQL
- 正式上线：PostgreSQL

### 7.2 为什么正式环境推荐 PostgreSQL

因为 PostgreSQL 更适合生产环境：

- 稳定
- 并发更好
- 备份恢复成熟
- 生态完善
- 后续可扩展

SQLite 适合本地开发和演示，但不适合正式多人使用场景。


### 7.3 关系型数据库要存什么

建议至少设计这些表：

#### users
- 用户信息
- 登录认证

#### documents
- 文档名称
- 文件路径
- 上传者
- 上传时间
- 解析状态
- 索引状态

#### document_chunks
- 可选
- 如果后续想自己管理 chunk 元数据，可存 chunk 信息

#### chat_sessions
- 会话信息

#### chat_messages
- 用户消息和模型消息

#### indexing_tasks
- PDF 解析/索引任务状态


### 7.4 向量数据库怎么处理

当前项目使用 Chroma，这是可以接受的，适合原型和单机部署。

#### 前期建议
先继续用 Chroma。

原因：

- 集成成本低
- 适合当前阶段
- 不需要一开始就搭重型基础设施

#### 中后期可选替代

如果文档量变大、检索需求更复杂，可以考虑：

- Qdrant
- Milvus
- pgvector（如果想和 PostgreSQL 更紧密集成）

#### 我的建议

对于你这个项目的发展路线：

- 第一阶段：Chroma
- 第二阶段：评估是否切到 Qdrant 或 pgvector


## 8. 文件存储怎么处理

PDF 本身不适合放数据库里。

建议存储方式：

- 本地开发：本地文件系统
- 服务器早期：服务器磁盘目录
- 正式稳定版：对象存储

对象存储可选：

- 阿里云 OSS
- 腾讯云 COS
- AWS S3
- Cloudflare R2

### 为什么后期建议对象存储

- 文件管理更可靠
- 容量扩展方便
- 可以做备份
- 可以与多台应用服务器解耦


## 9. 大模型和 Embedding 应该怎么选

### 9.1 当前方案

- 对话模型：DeepSeek API
- Embedding：本地 HuggingFace 模型

这个方案对原型期是可以的。

### 9.2 后续更推荐的设计

把模型调用做成“可替换提供商”结构。

也就是在代码层支持：

- DeepSeek
- OpenAI
- 未来其他模型提供商

原因：

- 避免被单一供应商绑定
- 方便比较效果和成本
- 方便做不同环境配置


### 9.3 Embedding 的建议

前期可以继续用本地模型，但要注意：

- 安装成本高
- 依赖重
- 服务器资源占用大

如果未来想要更稳定的线上服务，可考虑：

- 使用 API Embedding
- 或使用更轻、更稳定的本地 embedding 模型

## 11. 项目应该如何分阶段开发

这是最重要的一部分。

我建议整个项目分成 6 个阶段。


## 12. 第一阶段：把原型整理成“稳定开发版”

目标：

- 能稳定启动
- 前后端能联调
- 依赖完整
- 测试基本可跑
- 代码结构不再混乱

这个阶段要做的事：

1. 补全 `requirements.txt`
2. 修复测试脚本和当前实现不一致的问题
3. 整理 `.env.example`
4. 统一前后端接口配置
5. 修复当前文档选择和后端检索不绑定的问题
6. 增加基础启动文档
7. 增加最小可用日志

这是我们最应该先做的一阶段。


## 13. 第二阶段：建立真正的业务数据层

目标：

- 引入 PostgreSQL
- 建立 ORM 模型
- 建立迁移机制
- 让文档、任务、聊天记录真正入库

这个阶段要做的事：

1. 引入 SQLAlchemy 和 Alembic
2. 建立 documents、chat_sessions、chat_messages 等表
3. 上传 PDF 后把文档信息写入数据库
4. 索引状态可查询
5. 聊天记录可存储


## 14. 第三阶段：引入异步任务系统

目标：

- 上传不阻塞
- 大文件索引不拖垮接口
- 可以追踪任务状态

这个阶段要做的事：

1. 引入 Redis
2. 引入 Celery
3. PDF 解析和索引改成后台任务
4. 前端轮询或订阅任务状态


## 15. 第四阶段：升级前端为完整产品界面

目标：

- 不只是一个聊天页面
- 有文档管理
- 有知识库视角
- 有会话管理

这个阶段要做的事：

1. 引入 Vue Router
2. 引入 Pinia
3. 重构页面结构
4. 增加文档列表页、聊天页、设置页
5. 提升 UI 一致性


## 16. 第五阶段：上线部署

目标：

- 可在服务器稳定运行
- 可通过域名访问
- 支持日志、重启、备份

目标：

- 性能更好

1. 增加用户系统

## 18. 怎么上线到服务器

### 18.1 最推荐的上线架构

一开始建议用一台 Linux 云服务器即可，比如 Ubuntu 22.04。

部署结构：

- Nginx
- 前端静态文件
- FastAPI 后端容器
- PostgreSQL 容器
- Redis 容器
- 可选 Celery Worker 容器

### 18.2 为什么这样部署

原因是简单、清晰、成熟：

- Nginx 负责域名和反向代理
- 前端构建后直接由 Nginx 提供静态资源
- 后端 API 单独运行
- PostgreSQL 管业务数据
- Redis 管缓存和任务
- Worker 处理耗时索引

这套结构非常适合作为第一版正式上线方案。


## 19. Docker 应该怎么打包

### 19.1 为什么要 Docker

Docker 的核心价值是：

- 让开发环境和生产环境更一致
- 减少“我电脑能跑、服务器跑不了”的问题
- 上线更简单
- 易于迁移

### 19.2 这个项目未来建议拆成几个容器

建议至少拆成：

1. `frontend`
2. `backend`
3. `postgres`
4. `redis`
5. `worker`（后期）

### 19.3 前端容器

负责：

- 构建 Vue 项目
- 产出静态文件
- 由 Nginx 提供访问

### 19.4 后端容器

负责：

- 运行 FastAPI
- 处理 API 请求
- 调用 LLM
- 查询数据库和向量库

### 19.5 PostgreSQL 容器

负责：

- 业务数据持久化

### 19.6 Redis 容器

负责：

- 缓存
- 任务队列

### 19.7 Worker 容器

负责：

- PDF 索引
- 重建向量库
- 长耗时后台任务

## 22. 为什么不是一开始就上最复杂的方案

因为你现在最需要的不是“堆满技术名词”，而是建立一条能持续推进的开发路径。

如果一开始就上：

- 微服务
- Kubernetes
- 多环境流水线
- 分布式存储
- 重型权限系统

那大概率会直接把项目复杂度拉爆。

更好的策略是：

- 先单体应用
- 前后端分离
- 明确分层
- Docker 化
- 一台服务器先跑通
- 之后再慢慢增强

## 28. 因此，下一阶段最值得做什么

既然当前后端已经完成了前三阶段的大部分基础，接下来最值得投入的，不再是继续堆后端基础设施，而是把“产品体验层”补起来。

我建议后续优先级改成下面这样：

### 28.1 第一优先级：把前端补到能真正承接后端能力

现在后端已经有了文档、聊天、任务这些概念，前端也应该同步升级，而不是只保留一个简单聊天页。

这一阶段最值得做的是：

- 文档列表页
- 上传后的任务状态展示
- 文档处理中的轮询刷新
- 聊天会话列表
- 历史消息查看
- 错误状态提示

### 28.2 第二优先级：补全文档生命周期管理

要让系统更像正式产品，文档不能只有“上传”这一个动作。

还应该逐步补上：

- 删除文档
- 删除对应索引
- 重建索引
- 查看文档详情
- 区分处理中、成功、失败状态

### 28.3 第三优先级：增强 RAG 的可控性

当前系统已经能跑通 RAG，但如果想明显提升质量，后续要开始针对检索质量做优化。

后面可以逐步加入：

- 按文档过滤检索
- 更稳定的引用片段返回
- chunk 策略优化
- 检索结果去重
- rerank
- 更清晰的上下文拼装策略

### 28.4 第四优先级：再考虑用户系统和部署

用户、权限、项目空间、线上部署都很重要，但它们更适合放在“前后端产品链路已经顺滑”之后。

也就是说，下一步不一定要先把系统做得更“大”，而是要先把它做得更“完整”。


## 29. 现在推荐你怎么读这个仓库

如果你想真正建立工程感觉，不建议从零散文件乱看。更好的阅读顺序是：

6. 再看 `app/tasks/`

## 30. 现在这份总文档应该怎么使用
