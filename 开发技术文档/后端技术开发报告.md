# 后端技术开发报告

## 1. 文档目的

这份报告用于完整记录本阶段后端重构的目标、思路、实施过程、代码结构、技术选择、测试方法和后续规划。

它有三个用途：

1. 帮助你系统复盘这次开发过程
2. 作为后续继续迭代时的技术依据
3. 作为未来面试或项目介绍时的表达材料


## 2. 本阶段的开发目标

本阶段属于整个项目的第一阶段：把一个“能跑的原型后端”整理成“可以继续稳定开发的后端工程骨架”。

具体目标有：

- 清理临时脚本、失效文件和空壳目录
- 把后端代码从松散结构整理成明确分层
- 让接口、配置、服务职责更清晰
- 补足后端依赖声明
- 建立真正可运行的自动化测试
- 让项目具备持续开发的基础


## 3. 重构前的问题

在开始本次重构前，后端存在以下问题。

### 3.1 代码结构不清晰

原来的后端主要问题是功能堆叠在少数几个文件中：

- 路由和业务边界不够清楚
- 文件上传逻辑写在 `app/main.py`
- RAG 逻辑集中在单个 service 中
- 配置项较少，不利于扩展

### 3.2 临时脚本和测试混在根目录

原来根目录有大量形如：

- `test_llm.py`
- `test_rag.py`
- `test_stream_rag.py`
- `test_final_rag_chat.py`
- `test_chat_rag.py`
- `reindex.py`

这些文件更像手工验证脚本，而不是正式测试。

问题在于：

- 与 `pytest` 体系不兼容
- 有些函数名已经和当前实现不一致
- 不能作为长期工程质量保障

### 3.3 空壳目录过多

原项目里存在以下暂时没有实际业务价值的目录：

- `app/models/`
- `app/tasks/`
- `app/services/chains/`
- `app/services/tools/`

这些目录当前没有承载真实能力，反而会制造“架构好像很大，但其实还是空的”的假象。

### 3.4 后端测试环境不稳定

项目虚拟环境最开始连 `pip` 都不完整，`pytest` 也没有安装。

另外，Windows 当前环境对默认临时目录权限比较严格，直接使用 `pytest` 默认临时目录时会报权限错误。

### 3.5 业务逻辑仍处于原型阶段

旧代码中的典型问题包括：

- 上传接口包含无意义调试代码
- 索引逻辑和向量库访问耦合较高
- 文档检索缺少按文档名过滤的清晰入口
- 配置项不足，不适合继续演化


## 4. 本次重构的总体设计思路

本次后端重构遵循了一个非常重要的原则：

**只做当前阶段真正需要的工程化整理，不提前引入数据库、任务队列等下一阶段能力。**

所以本次重点不是“做全功能平台”，而是“把基础骨架搭稳”。

最终采用的后端分层如下：

- `app/main.py`
  - 应用入口
- `app/core/`
  - 配置和日志
- `app/api/`
  - 路由层
- `app/schemas/`
  - 请求/响应模型
- `app/services/`
  - 业务逻辑
- `tests/`
  - 自动化测试

## 6. 每层职责说明

### 6.1 `app/main.py`

这个文件现在只负责应用装配：

- 创建 FastAPI 应用
- 注册 CORS
- 挂载 API 路由
- 在生命周期中初始化日志和存储目录

这样做的好处是入口文件非常干净，后续好维护。


### 6.2 `app/core/`

#### `config.py`

负责所有核心配置，包括：

- 项目名
- API 前缀
- 模型配置
- 向量库目录
- PDF 存储目录
- CORS
- 切块参数
- 检索数量

这使得后续调整参数不必改业务代码。

#### `logging.py`

当前先做最小可用日志配置，为后续接入更完整日志系统留出位置。


### 6.3 `app/api/`

这一层只做接口输入输出组织，不承载复杂业务逻辑。

#### `routes/health.py`

提供健康检查接口：

- `GET /api/v1/health`

#### `routes/documents.py`

负责文档相关接口：

- `GET /api/v1/files`
- `POST /api/v1/files/upload`

#### `routes/chat.py`

负责问答接口：

- `POST /api/v1/stream_chat`

它通过 SSE 将模型输出流式返回前端。


### 6.4 `app/schemas/`

这一层负责定义请求和响应的数据结构。

意义是：

- 接口字段统一
- 便于前后端联调
- 便于后续文档生成
- 降低低级字段错误


### 6.5 `app/services/`

这一层是业务核心。

#### `document_service.py`

负责：

- 确保存储目录存在
- 安全处理上传文件名
- 保存 PDF 文件
- 列出已上传文档

#### `indexing_service.py`

负责：

- 读取 PDF
- 按设置切块
- 标准化元数据
- 将 chunk 写入向量库

#### `vector_store.py`

负责：

- 初始化本地 embedding 模型
- 创建 Chroma 向量库实例

#### `llm_service.py`

负责：

- 创建 DeepSeek 对话模型客户端

#### `rag_service.py`

负责：

- 根据问题执行向量检索
- 支持按 `source_name` 过滤
- 构造上下文
- 调用大模型并流式返回结果


## 7. 关键重构决策与原因

### 7.1 为什么要删除根目录那些旧测试脚本

因为它们的问题不只是“风格不好”，而是已经开始伤害工程结构：

- 有的依赖旧函数名
- 有的不是标准测试
- 有的只适合人工运行

继续保留它们，会让项目同时存在两套质量标准：

- 一套是临时脚本
- 一套是未来真正的自动化测试

这会让项目越来越乱。


### 7.2 为什么现在不引入数据库

虽然数据库是下一阶段必须引入的，但这一阶段不急着上。

原因是：

- 当前最核心问题不是“没有数据库”
- 而是“项目骨架还不稳”
- 如果现在同时引入 ORM、迁移、业务表，会把重构复杂度放大

正确顺序应该是：

1. 先把后端接口和服务层理顺
2. 再引入数据库模型


### 7.3 为什么保留 Chroma

Chroma 目前足够支持本阶段目标：

- 集成简单
- 对当前项目规模足够
- 能快速验证 RAG 主链路

所以当前阶段没有必要为了“更专业”而立刻切换更重的向量数据库。


### 7.4 为什么要补 `schemas`

对没有经验的开发者来说，很多时候容易把“能传数据”误以为“接口设计完成了”。

但真正长期开发时，数据结构必须被显式定义。

引入 `schemas` 的好处：

- 后续前端调接口更明确
- 请求字段更规范
- 错误更早暴露
- 后续接入数据库更顺滑


### 7.5 为什么要把路由拆成多个文件

随着业务增加，一个单文件路由很快会变得不可维护。

拆成：

- 健康检查
- 文档接口
- 聊天接口

是最自然的最小分法。

## 9. 接口设计结果

当前后端接口如下：

### 9.1 健康检查

```http
GET /api/v1/health
```

返回：

```json

```


### 9.2 获取文档列表

```http
GET /api/v1/files
```

返回：

```json

  "documents": ["a.pdf", "b.pdf"]

```


### 9.3 上传 PDF

```http
POST /api/v1/files/upload
```

返回：

```json
{
  "status": "success",
  "filename": "paper.pdf",
  "chunks_indexed": 4,
  "message": "File uploaded and indexed successfully"
}
```


### 9.4 流式问答

```http
POST /api/v1/stream_chat
```

请求体：

```json
{
  "message": "What is the main method?",
  "source_name": "paper.pdf"
}
```

说明：

- `source_name` 是可选字段
- 当传入后，检索会优先过滤到指定文档


## 10. 本阶段测试设计

这次我们不再依赖手工脚本，而是改成自动化测试。

### 10.1 新增的测试内容

#### `test_health_api.py`

验证健康检查接口正常返回。

#### `test_chat_api.py`

通过 monkeypatch 模拟 RAG 流输出，验证：

- SSE 响应头正确
- 流数据格式正确
- `[DONE]` 结束标记存在

#### `test_documents_api.py`

验证：

- 文档列表接口正常返回
- 上传接口能返回标准结果
- 文件保存时会自动清洗危险路径，避免目录穿越

#### `test_rag_service.py`

验证：

- `retrieve_context()` 在指定 `source_name` 时会正确加上过滤条件
- 生成的上下文包含来源和页码


### 10.2 为什么测试里用了 monkeypatch

因为这一阶段测试的目标不是去真实调用：

- DeepSeek
- Chroma
- PDF 解析器

而是验证后端自身的接口契约和服务逻辑。

这样可以让测试：

- 快
- 稳
- 不依赖外部网络
- 不依赖真实模型返回


## 11. 测试环境问题与解决过程

这是这次开发里一个很典型的工程问题，也很适合面试时讲。

### 11.1 问题一：虚拟环境没有 pip

最开始项目虚拟环境不完整，执行检查时发现：

- `pip` 缺失
- `pytest` 未安装

解决方案：

1. 使用 `ensurepip` 恢复 `pip`
2. 安装 `pytest`


### 11.2 问题二：Windows 临时目录权限导致 pytest 报错

原本使用 `pytest` 内置临时目录机制时，Windows 环境对：

- 系统临时目录
- pytest 清理目录

都出现了权限问题。

解决方案：

- 不依赖 `pytest` 的 `tmp_path` 默认机制
- 改为在项目内手动建立 `tests/.tmp` 目录
- 测试前创建，测试后删除

这是一个很典型的“环境问题不等于业务代码错误”的案例。

## 14. 从面试表达角度如何讲这次重构

如果以后你在面试中介绍这次工作，可以这样组织表达：

### 14.1 项目背景

我在一个基于 FastAPI 和 Vue 的 RAG 项目中，负责把早期原型整理成可持续开发的工程骨架。

### 14.2 我发现的问题

- 原型期代码职责混杂

- 重构后端为 `api / schemas / services / core` 分层

- 给上传、流式问答、检索过滤建立自动化测试
- 补齐依赖声明和环境修复流程

### 14.4 结果

- 后端结构更清晰

## 15. 当前仍然没有做的事情

不是因为这些不重要，而是因为它们属于下一阶段。

## 17. 本阶段总结

这次后端重构的价值，不在于增加了多少花哨功能，而在于完成了一个真正重要的转变：

**把一个原型后端，整理成了一个能继续认真开发的后端。**

它为后续所有工作提供了基础：

- 更稳的目录结构
- 更清晰的职责边界
- 更标准的接口设计
- 更可靠的测试方式
- 更合理的依赖管理

这一步看起来不如“新功能”显眼，但它决定了项目后面能不能越做越顺。
