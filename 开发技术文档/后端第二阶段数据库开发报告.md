# 后端第二阶段数据库开发报告

## 1. 文档目的

这份文档记录第二阶段后端开发的完整过程。  
本阶段的主题是：

**把后端从“只有文件和向量库的原型系统”，升级为“具备真实业务数据层的后端系统”。**

这份文档的用途有三个：

1. 帮你系统学习后端数据库接入过程
2. 作为后续继续开发时的技术依据
3. 作为未来面试时介绍项目演进的讲解材料


## 2. 本阶段目标

第一阶段完成后，项目已经有了较清晰的后端分层，但仍然缺少真正的业务数据层。

因此第二阶段的目标是：

- 引入 SQLAlchemy ORM
- 建立数据库连接与会话管理
- 设计并创建文档和聊天相关数据表
- 让文件上传流程落库
- 让聊天会话和消息落库
- 提供查询聊天会话和消息的接口
- 引入 Alembic 迁移骨架
- 为下一阶段用户系统和异步任务系统打基础


## 3. 第二阶段之前的问题

在开始这轮开发前，系统虽然可以上传 PDF 和进行问答，但存在几个核心缺口。

### 3.1 文档没有真正入库

上传 PDF 后，文件被保存到了磁盘，也被写入了向量库，但数据库里没有任何文档记录。

这意味着：

- 无法管理文档状态
- 无法记录索引成功或失败
- 无法做后续的文档列表、删除、重建索引
- 无法在产品层面形成“文档实体”


### 3.2 聊天没有会话概念

虽然前端能发送问题并收到回答，但后端并没有持久化：

- 会话
- 用户消息
- 助手消息

所以系统没有聊天历史，也不可能支撑：

- 会话列表
- 历史消息
- 多轮对话产品化


### 3.3 只有文件系统和向量库，没有业务数据库

一个正式产品通常至少有两层数据：

1. 业务数据库
2. 检索数据库

而之前这个项目只有检索相关的数据和本地文件，没有业务层的数据抽象。


## 4. 本阶段总体设计思路

本阶段没有直接一步冲到“完整企业级后端”，而是采用了一个适合当前项目阶段的设计：

- 业务数据库：SQLite（开发阶段）
- ORM：SQLAlchemy 2.x
- 迁移工具：Alembic
- 保持单体后端结构，不拆服务
- 先完成最关键的数据实体：文档、聊天会话、聊天消息

这样设计的原因是：

- 你现在需要先建立完整的数据流思维
- 不应该在这一阶段引入过多复杂度
- 但也不能继续停留在“只有文件，没有业务数据”的原型状态

## 6. 本阶段的数据库模型设计

本阶段共设计了三张核心表。


## 7. `documents` 表设计

### 7.1 作用

`documents` 表负责记录每一份上传文档的业务元数据。

### 7.2 字段

- `id`
- `filename`
- `file_path`
- `status`
- `chunk_count`
- `last_error`
- `created_at`
- `updated_at`
- `indexed_at`

### 7.3 为什么这么设计

#### `filename`
用于在业务层标识文档。

#### `file_path`
记录 PDF 在磁盘中的位置。

#### `status`
表示文档处理状态，例如：

- `uploaded`
- `ready`
- `failed`

这对后续前端展示和后台任务状态都很重要。

#### `chunk_count`
记录索引生成了多少 chunk。

#### `last_error`
如果索引失败，可以保留错误信息，方便排查。

#### `indexed_at`
记录真正索引完成的时间。


## 8. `chat_sessions` 表设计

### 8.1 作用

用来表示一次聊天会话。

### 8.2 字段

- `id`
- `title`
- `source_name`
- `created_at`
- `updated_at`

### 8.3 为什么需要这张表

如果没有 `chat_sessions`，所有消息都只是零散文本，无法形成产品层面的“会话”概念。

有了这张表之后，后续才能支持：

- 会话列表
- 会话恢复
- 按会话查看历史消息
- 一个会话绑定一个文档或知识源


## 9. `chat_messages` 表设计

### 9.1 作用

记录会话中的每一条消息。

### 9.2 字段

- `id`
- `session_id`
- `role`
- `content`
- `source_name`
- `created_at`

### 9.3 为什么这么设计

#### `session_id`
把消息归属到某个会话。

#### `role`
区分：

- 用户消息
- 助手消息

#### `content`
保存真正的文本内容。

#### `source_name`
为后续多文档、多知识库场景保留来源信息。


## 10. 为什么当前阶段没有更多表

这次我们刻意没有继续往下加：

- 用户表
- 权限表
- 知识库表
- 后台任务表

原因不是它们不重要，而是这些属于下一阶段更合理。

当前阶段最重要的是先把：

- 文档实体
- 聊天会话
- 聊天消息

这三层打通。


## 11. 业务流程是怎么接到数据库上的

这部分是第二阶段最关键的地方。


## 12. 文件上传流程的变化

### 12.1 改造前

旧流程是：

1. 上传 PDF
2. 保存到文件系统
3. 写入向量库
4. 返回成功

问题是：

- 没有文档记录
- 不知道索引状态
- 无法追踪失败


### 12.2 改造后

现在的流程是：

1. 上传 PDF
2. 保存文件到磁盘
3. 在 `documents` 表中创建或更新文档记录
4. 执行 PDF 索引
5. 索引成功则更新状态为 `ready`
6. 索引失败则更新状态为 `failed` 并写入错误信息


### 12.3 这样改的好处

- 文档状态清晰
- 接口行为更像正式产品
- 失败可追踪
- 为后续重建索引、文档列表等功能打基础


## 13. 聊天流程的变化

### 13.1 改造前

原来的聊天流程是：

1. 接收问题
2. 检索
3. 调模型
4. SSE 返回流式结果

问题是：

- 没有聊天会话
- 没有消息存档
- 用户每次聊天在系统看来都是一次性请求


### 13.2 改造后

现在的流程是：

1. 接收消息和可选的 `session_id`
2. 如果没有 `session_id`，创建新的 `chat_session`
3. 先把用户消息保存到 `chat_messages`
4. 调用 RAG 流式生成回答
5. 一边流式返回，一边在内存中累计回答文本
6. 回答完成后，把助手消息完整保存到数据库
7. 在响应头中返回 `X-Session-Id`


### 13.3 为什么这样设计

这是兼顾“前端兼容”和“后端产品化”的一个折中方案。

优点：

- 不破坏现有流式接口
- 新增会话能力
- 可以无缝接入前端历史记录
- 后续支持多轮对话很自然


## 14. 新增的聊天相关接口

为了让聊天持久化真正可用，这次新增了几个接口。

### 14.1 创建会话

```http
POST /api/v1/chat/sessions
```

作用：

- 主动创建一个聊天会话


### 14.2 获取会话列表

```http
GET /api/v1/chat/sessions
```

作用：

- 返回当前系统里的聊天会话列表


### 14.3 获取单个会话及消息

```http
GET /api/v1/chat/sessions/{session_id}
```

作用：

- 返回某个会话及其所有消息


### 14.4 继续保留流式接口

```http
POST /api/v1/stream_chat
```

变化：

- 现在支持 `session_id`
- 会自动落库
- 返回 `X-Session-Id`


## 15. 本阶段对配置系统的改造

## 16. 引入 Alembic 的原因

这是第二阶段一个很重要的工程动作。

### 16.1 为什么不能永远靠 `create_all()`

`create_all()` 适合：

- 本地开发早期
- 快速启动项目

但它不适合长期生产演进，因为：

- 无法记录版本历史
- 无法回滚
- 无法多人协作跟踪结构变化


### 16.2 Alembic 在项目里的作用

Alembic 负责：

- 记录数据库结构变更
- 生成迁移脚本
- 按版本升级/回退数据库


### 16.3 本阶段的处理方式

本阶段采用“过渡式方案”：

- 启动时仍保留 `init_db()`，便于本地开发快速起步
- 同时加入 Alembic 骨架和第一份初始迁移

这样做的好处是：

- 学习成本低
- 开发体验不受影响
- 后续正式迁移已经有落点


## 17. 为什么初始迁移要手工修正

这次开发里有个很值得学习的点：

我最初先运行了 `init_db()` 创建数据库表，然后才执行 Alembic 自动生成迁移。  
因为数据库已经存在这些表了，所以 Alembic 自动生成出来的是一个空迁移。

这其实是很典型的工程现象：

- 自动工具不会替你纠正开发顺序
- 如果上下文不对，工具就会产出不符合预期的结果

所以最终我把初始迁移改成了真正的建表迁移。

这也是一个很适合面试里讲的细节：

**你不仅会用工具，还会判断工具生成的结果是否正确。**

## 21. 为什么测试需要单独处理 SQLite 文件

这是这次阶段里一个比较真实的工程坑。

在 Windows 环境下，SQLite 数据库文件会被 SQLAlchemy engine 持有。  
如果测试之间复用同一个临时数据库文件，很容易在删除目录时碰到：

- 文件被占用
- 无法删除
- 测试之间互相污染

最后的解决方式是：

- 每个测试用例使用独立临时目录
- 每个测试用独立 SQLite 文件
- teardown 时主动 `dispose` engine

这是很标准的“测试环境隔离”处理思路。

这次改造的价值非常大，因为系统从这一刻开始终于有了“产品化数据层”。

## 25. 从面试表达角度怎么讲这一阶段

如果以后你在面试中讲这个阶段，可以这样表达：

### 项目背景

我接手的是一个已经能进行 PDF 问答的 RAG 原型，但它缺少真正的业务数据库层。

### 问题

- 上传的文档没有业务记录
- 聊天没有会话和消息持久化
- 系统没有数据库迁移能力

### 我的方案

- 引入 SQLAlchemy 2.x
- 设计 `documents`、`chat_sessions`、`chat_messages` 三张核心表
- 重构上传和聊天接口，让其接入数据库
- 保持流式问答接口兼容，同时增加聊天历史能力
- 引入 Alembic 并建立初始迁移
- 用自动化测试验证数据库落库和接口行为

### 成果

- 文档和聊天能力正式产品化
- 迁移和数据演进能力建立
- 为用户系统和异步任务系统奠定基础


## 26. 当前仍然没有做的事情

到第二阶段结束时，我们仍然没有做：

- 用户登录与鉴权
- 文档权限隔离
- 异步索引任务
- Redis / Celery
- 文档删除与重建索引 API
- 多知识库支持
- PostgreSQL 正式切换

这些内容都很重要，但应放在后续阶段逐步引入。


## 27. 下一阶段建议

数据库层已经接好了，接下来最合理的第三阶段方向是：

**引入异步任务系统和文档处理状态流转。**

也就是：

1. 引入 Redis
2. 引入 Celery
3. 把 PDF 索引从同步接口中拆出去
4. 上传接口只负责创建文档和任务
5. 前端轮询或订阅任务状态

这样一来，系统就会从“同步原型”升级为“更接近正式生产行为的 AI 文档平台”。


## 28. 本阶段总结

第二阶段最核心的成果，不是“加了一些表”，而是完成了一个本质升级：

**后端从无状态原型，升级成了有业务数据模型的系统。**

从这一阶段开始，这个项目已经不再只是一个简单的 demo，而是具备了继续成长为完整产品的真正基础。
