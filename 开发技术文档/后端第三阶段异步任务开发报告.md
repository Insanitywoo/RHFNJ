# 后端第三阶段异步任务开发报告

## 1. 文档目的

这份文档记录后端第三阶段的开发过程。  
本阶段的主题是：

**把文档索引从同步请求流程中拆出来，升级为可追踪、可扩展的异步任务系统。**

这份文档主要用于：

1. 帮你理解为什么正式系统不能长期依赖同步耗时任务
2. 记录本阶段的架构设计和代码实现
3. 作为以后面试讲述“后端异步化改造”的素材


## 2. 本阶段目标

第二阶段结束后，系统已经有了数据库层，但上传 PDF 仍然会在请求里直接做索引。

这会带来几个问题：

- 上传接口耗时长
- 大 PDF 会拖慢请求
- 如果索引失败，请求体验很差
- 前端无法明确知道“任务正在处理还是已经失败”
- 后续如果要扩展为多用户或高并发，会很不稳

因此第三阶段的目标是：

- 引入异步任务架构
- 为文档索引建立任务记录表
- 将上传流程改为“创建任务 + 派发任务”
- 增加任务状态查询接口
- 保留数据库层的状态追踪能力
- 引入 Celery / Redis 集成入口
- 保证测试和迁移都同步更新


## 3. 为什么需要异步任务

这是一个很核心的工程问题。

### 3.1 同步索引的本质问题

原来上传流程是：

1. 上传 PDF
2. 保存文件
3. 立即解析 PDF
4. 立即切块
5. 立即向量化
6. 立即写入向量库
7. 请求返回

这个模式在 demo 阶段可以接受，但在正式系统里会有明显缺点：

- 用户要一直等
- 请求容易超时
- 大文件体验差
- 耗时任务挤占 API 进程
- 无法很好地管理失败重试


### 3.2 更合理的正式流程

更合理的模式应该是：

1. 用户上传 PDF
2. 后端先保存文件
3. 后端创建数据库里的文档记录
4. 后端创建数据库里的索引任务记录
5. 后端把任务派发给后台 worker
6. API 很快返回 `task_id`
7. 前端后续轮询任务状态
8. worker 在后台完成真正索引

这就是本阶段实现的核心思想。


## 4. 本阶段总体架构设计

本阶段在原有后端基础上新增了三个核心层次：

1. 任务模型层
2. 任务服务层
3. Celery 任务执行层

同时保留原有文档和数据库结构，使整个系统形成下面的链路：

```text
上传请求
  -> 保存文件
  -> 创建 Document
  -> 创建 IndexingTask
  -> 派发 Celery 任务
  -> 返回 task_id

后台 Worker
  -> 读取任务
  -> 更新任务状态为 running
  -> 执行 PDF 索引
  -> 更新 Document 状态
  -> 更新 Task 状态
```


## 5. 本阶段新增的关键数据模型

### 5.1 `indexing_tasks` 表

这是第三阶段最关键的数据结构。

新增字段包括：

- `id`
- `document_id`
- `task_type`
- `status`
- `celery_task_id`
- `message`
- `error`
- `created_at`
- `started_at`
- `finished_at`


### 5.2 为什么需要这张表

即使 Celery 自己能记录任务结果，正式产品仍然应该在业务数据库里有自己的任务表。

原因是：

- 业务系统不能完全依赖队列系统做唯一状态来源
- 任务状态需要和文档状态强关联
- 前端更适合读业务数据库抽象出来的状态
- 以后可以做任务列表、失败重试、审计记录


### 5.3 `status` 的设计意义

本阶段使用的任务状态包括：

- `queued`
- `running`
- `succeeded`
- `failed`

这是最小但非常实用的一套状态机。

它能支撑：

- 前端显示排队中
- 前端显示处理中
- 完成后提示成功
- 失败后展示错误


## 6. 文档状态流转的升级

第三阶段不仅增加了任务状态，也升级了文档状态管理。

### 6.1 文档状态不再只是一种完成结果

现在文档状态会经历：

- `uploaded`
- `queued`
- `processing`
- `ready`
- `failed`


### 6.2 为什么文档状态和任务状态都要存在

这是一个很典型的设计点：

- 任务状态描述的是“后台任务执行情况”
- 文档状态描述的是“这个文档当前处于什么业务阶段”

举例：

- 一个文档可能当前状态是 `processing`
- 与此同时它对应的索引任务状态是 `running`

两者相关，但不是一个概念。


## 7. Celery 集成设计

### 7.1 为什么选 Celery

Celery 是 Python 生态里非常常见的异步任务队列方案，适合：

- 文档处理
- 后台索引
- 批量任务
- 定时任务

它的优势是：

- 生态成熟
- 和 Python 后端集成简单
- 非常适合这种 AI / 文档处理场景


### 7.2 为什么需要 Redis

Redis 在这里主要承担：

- Celery broker
- Celery result backend

简单理解就是：

- broker 负责收任务
- backend 负责保存结果状态


### 7.3 本阶段的做法

本阶段没有直接把 Redis 和 Worker 运行在当前对话里长期挂着，而是先把：

- Celery 应用配置
- 异步任务函数
- 配置项
- 数据库任务追踪

全部接好。

也就是说，这一阶段完成的是：

**异步任务架构接入与代码闭环**

而不是“已经在生产机器上长期跑 worker”。

## 9. 上传流程改造前后对比

### 9.1 改造前

```text
Upload API
  -> save file
  -> index_pdf()
  -> return success
```

### 9.2 改造后

```text
Upload API
  -> save file
  -> create/update document
  -> mark document queued
  -> create indexing task
  -> dispatch celery task
  -> return queued + task_id
```


### 9.3 改造后的价值

- 接口响应更快
- 业务状态更清晰
- 前端可追踪任务进度
- 后续容易扩展失败重试机制


## 10. 新增接口设计

第三阶段新增了与任务相关的接口。

### 10.1 查询单个任务状态

```http
GET /api/v1/files/tasks/{task_id}
```

作用：

- 返回某个索引任务的完整状态


### 10.2 查询某个文档下的任务列表

```http
GET /api/v1/files/{document_id}/tasks
```

作用：

- 返回某份文档相关的所有索引任务


### 10.3 上传接口返回的变化

`POST /api/v1/files/upload` 不再同步返回索引完成结果，而是返回：

- `status = queued`
- `document_id`
- `task_id`

这意味着：

- 上传成功不等于索引完成
- 后续需要通过任务接口继续追踪状态


## 11. 为什么上传接口必须返回 `task_id`

因为异步系统的关键不只是“后台做了”，而是：

**前端必须知道去追踪哪一个后台任务。**

如果没有 `task_id`，前端就只能：

- 反复刷新整个文档列表
- 靠猜测文档状态变化

这会让产品体验很差。


## 12. 测试设计升级

第三阶段测试重点不再只是验证接口返回，而是验证：

- 任务被正确创建
- 文档状态被正确更新
- worker 执行后状态流转正确
- Alembic 迁移链可用


## 13. 上传接口测试

这一阶段上传接口测试做了这些验证：

- 返回 `queued`
- 返回 `task_id`
- 数据库里生成了 `IndexingTask`
- `Document.status` 变成 `queued`
- 任务状态是 `queued`
- 任务查询接口可正常返回结果
- 文档任务列表接口可正常返回结果


## 14. Worker 执行测试

这次新增了一个很重要的测试：

`test_run_document_indexing_updates_document_and_task`

它验证的是：

1. 已存在的文档和任务
2. worker 执行索引任务
3. 文档状态更新为 `ready`
4. 文档 `chunk_count` 被正确写入
5. 任务状态更新为 `succeeded`

这是第三阶段最能体现“异步任务真正落地”的测试。

同时本次复核还补充了失败路径测试：

`test_run_document_indexing_marks_failures`

它验证的是：

1. worker 执行过程中如果索引抛错
2. 文档状态会更新为 `failed`
3. 文档 `last_error` 会记录错误原因
4. 任务状态会更新为 `failed`
5. 任务 `error` 字段会保留失败信息

这样第三阶段不只是验证“成功时能跑通”，也验证了“失败时状态是否可追踪”。

## 18. 为什么这一步很有产品意义

很多初学者会觉得“异步任务只是后端技术细节”，其实不是。

它直接影响产品体验：

- 上传后是否要等很久
- 前端能不能显示处理中
- 失败能不能看懂
- 系统能不能稳定处理更大文件

所以第三阶段的价值不是“加了 Celery”，而是：

**让产品开始拥有可靠的后台处理能力。**


## 19. 从面试角度如何讲这次改造

你可以这样讲：

### 背景

项目原本在上传 PDF 时同步做索引，导致上传接口耗时长，不利于扩展。

### 问题

- 请求线程被长任务占用
- 文档处理状态不可追踪
- 前端无法知道任务是否真正完成

### 我的方案

- 引入 Celery / Redis 异步任务架构
- 设计 `indexing_tasks` 业务表
- 重构上传接口为“创建任务 + 派发任务”
- 增加任务状态接口
- 用自动化测试验证任务流转
- 补充 Alembic 迁移

### 成果

- 文档索引流程异步化
- 文档和任务状态可追踪
- 接口更符合正式产品行为
- 为后续失败重试和任务管理打下基础
