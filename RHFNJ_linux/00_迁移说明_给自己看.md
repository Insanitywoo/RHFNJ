# RHFNJ Linux 迁移说明

这份说明的目标是帮助你把当前 Windows 下的 `RHFNJ` 项目迁移到 WSL2 的 Ubuntu 环境，并在那里继续开发。

你现在拿到的 `RHFNJ_linux` 文件夹，已经是一个适合复制到 Ubuntu 的“可移植开发包”。


## 1. 这个文件夹里已经包含了什么

这里已经保留了后续继续开发真正需要的内容：

- 后端源码：`app/`
- 数据库迁移：`alembic/`、`alembic.ini`
- 前端源码：`frontend/src/`
- 前端静态资源与配置：`frontend/public/`、`frontend/package.json` 等
- 测试：`tests/`
- 项目说明：`README.md`
- 总技术文档：`技术文档.md`
- 分阶段开发报告：`开发技术文档/`
- Docker / Compose / Nginx / Dockerfile 等部署文件
- 示例 PDF 和当前 `data/papers/` 里的论文文件

这些内容都可以直接复制到 Ubuntu 里继续使用。


## 2. 哪些内容没有被复制过去

下面这些内容都**不应该**直接从 Windows 搬到 Ubuntu，因为它们高度依赖当前系统环境，或者本来就是临时生成物：

- `.git/`（说明见下）
- `.venv/`
- `frontend/node_modules/`
- `frontend/dist/`
- `.pytest_tmp/`
- `pytest-cache-files-*`
- `.ruff_cache/`
- `rhfnj.db`
- `data/vector_store/`
- `data/vector_store_v2/`
- 根目录 `.env`

原因分别是：

### 2.0 为什么 `.git/` 没有放进 `RHFNJ_linux`

不是因为 Git 历史不能迁移，而是因为当前这个目录是在原仓库内部新建的。

如果把 `.git/` 也复制进去，会在当前 Windows 仓库里形成一个嵌套 Git 仓库，后续会很混乱。

所以这里采取的是更稳妥的策略：

- `RHFNJ_linux` 负责携带源码、配置和文档
- Git 历史你可以在 Ubuntu 里用 `git clone` 原 GitHub 仓库重新获取

如果你后面只是想在 Ubuntu 里继续开发，最推荐的方式是：

1. 把 `RHFNJ_linux` 当作“离线可迁移开发包”
2. 如果需要 Git 历史，再在 Ubuntu 里重新 `git clone`

### 2.1 `.venv/` 不能直接带过去

因为虚拟环境里有：

- Windows 平台的 Python 可执行文件
- Windows 平台的依赖二进制
- Windows 路径记录

这些到 Ubuntu 上都会失效。

### 2.2 `frontend/node_modules/` 不能直接带过去

因为 Node 包里会包含：

- 当前系统平台相关的可执行文件
- 构建缓存
- 依赖树里的平台差异

所以到了 Ubuntu 里必须重新执行 `npm install`。

### 2.3 `rhfnj.db` 不建议直接带过去

这不是因为 SQLite 一定不能跨平台，而是因为：

- 当前开发已经准备转向更规范的部署流程
- Docker Compose 里已经使用 PostgreSQL
- 与其继续沿用一份 Windows 下的 SQLite 开发库，不如在 Ubuntu 环境里重新初始化数据库

### 2.4 `data/vector_store/` 不建议直接带过去

向量库虽然本质上也是文件，但里面记录的是当前本机下生成的索引状态。

继续迁移过去容易带来两个问题：

- 你不容易判断索引是否和当前数据库一致
- 后面换环境后调试会变复杂

更稳妥的方式是：

- 把论文原文件带过去
- 在 Ubuntu 里重新索引

### 2.5 `.env` 没有复制

这是故意的，因为 `.env` 里可能包含：

- 真实 API Key
- 个人环境配置

这些不应该作为迁移包的一部分。

你需要在 Ubuntu 里根据 `.env.example` 重新创建新的 `.env`。


## 3. 到 Ubuntu 里以后需要重新安装或重新配置什么

这是最重要的一部分。

### 3.1 Python 虚拟环境

你需要在 Ubuntu 里重新创建：

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

### 3.2 前端依赖

你需要在 Ubuntu 里重新安装：

```bash
cd frontend
npm install
```

### 3.3 环境变量

你需要重新创建：

```bash
cp .env.example .env
```

然后按你的实际情况填写：

- `DEEPSEEK_API_KEY`
- `OPENAI_API_KEY`（可选）
- 其他数据库、Redis、文件目录配置

### 3.4 数据库迁移

如果你继续本地开发，建议直接重新执行迁移：

```bash
alembic upgrade head
```

### 3.5 向量索引

把 `data/papers/` 里的论文带过去后，建议重新上传或重新重建索引，而不是直接沿用旧的 Windows 向量库文件。

### 3.6 Docker 镜像和容器

Docker 本身不需要复制什么目录过去。

只要 Ubuntu 环境里 Docker / Docker Compose 可用，就可以直接在项目目录里重新构建：

```bash
docker compose up --build
```


## 4. 到 Ubuntu 后建议的开发顺序

建议你按这个顺序继续：

1. 把 `RHFNJ_linux` 文件夹复制到 Ubuntu
2. 进入项目目录
3. 创建新的 `.venv`
4. 安装 Python 依赖
5. 安装前端依赖
6. 新建 `.env`
7. 执行 `alembic upgrade head`
8. 先跑 `pytest`
9. 再跑前端构建
10. 最后跑 Docker Compose

如果你按这个顺序做，定位问题会容易很多。


## 5. 当前最推荐的 Ubuntu 目录结构

建议你把项目放在类似下面的位置：

```bash
~/projects/RHFNJ
```

或者：

```bash
/home/<your-user>/projects/RHFNJ
```

不要继续放在 Windows 挂载路径里，例如 `/mnt/c/...` 下做长期开发。

原因是：

- Linux 下文件性能更好
- Python / Node / Docker 的体验更稳定
- 更接近以后真实服务器环境


## 6. 到 Ubuntu 后怎么判断迁移成功

你可以用下面几个标准判断：

### 6.1 后端测试通过

```bash
pytest -q
```

### 6.2 前端类型检查和构建通过

```bash
cd frontend
npm run build
```

### 6.3 Docker Compose 能正常启动

```bash
docker compose up --build
```

### 6.4 前端能访问

浏览器打开：

- `http://localhost:8080`

### 6.5 后端健康检查正常

访问：

- `http://localhost:8080/api/v1/health`


## 7. 一句话理解这个迁移包

你可以把 `RHFNJ_linux` 理解成：

**只带走源码、配置、文档和论文原始文件，不带走任何平台相关缓存和生成物。**

这才是最适合跨 Windows -> Ubuntu 迁移的方式。
